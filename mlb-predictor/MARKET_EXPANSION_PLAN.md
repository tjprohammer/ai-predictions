# Market Expansion & Model Improvement Plan

**Status:** Planning  
**Goal:** Expand market coverage, add umpire/arsenal data, improve hit rate on player props and game totals.

---

## Phase 1 — New Odds API Markets (Zero New Data Sources)

All items below only require changes to `src/ingestors/market_totals.py`. The ingestion pipeline, DB tables (`game_markets`, `player_prop_markets`), and upsert logic already handle these market types.

### 1A. Game-Level Markets — add to `ODDS_API_GAME_PERIOD_MARKETS`

| Odds API Key | Internal `market_type` | Why |
|---|---|---|
| `team_totals` | `home_team_total` / `away_team_total` | Model already outputs `home_expected_runs` + `away_expected_runs` — this is the direct market for those numbers. **Highest priority.** |
| `spreads` | `run_line` | Run-line edge flows from our expected-run split; -1.5/+1.5 is exploitable when model projects 4+ run margin |
| `h2h` | `moneyline` | Win probability from run split; useful as parlay context and for F5 ML comparison |
| `h2h_1st_5_innings` | `first_five_moneyline` | Sharp angle when one SP dominates; pair with existing F5 total prediction |
| `alternate_totals` | `alt_game_total` | Shows sliding price at each alternate line; useful when model projection diverges far from consensus |

**Implementation:** In `market_totals.py`, add these keys to `ODDS_API_GAME_PERIOD_MARKETS`. Update `_extract_odds_api_event_first5_rows` (or add a sibling extractor) to parse team totals, spreads, and moneylines from the same per-event API call. **No extra API requests needed** — each event call already returns all markets.

### 1B. Player Prop Markets — add to `ODDS_API_PLAYER_PROP_MARKETS`

| Odds API Key | Internal `market_type` | Why |
|---|---|---|
| `batter_total_bases` | `batter_total_bases` | **Highest priority.** xwOBA, xSLG, ISO, HH% from existing features directly predict TB. |
| `batter_runs_scored` | `batter_runs_scored` | Lineup slot + OBP + game total → modellable from existing features |
| `batter_rbis` | `batter_rbis` | Runners-on-base context + power profile |
| `pitcher_hits_allowed` | `pitcher_hits_allowed` | Starter WHIP blended + opposing lineup xwOBA → direct prediction |
| `pitcher_earned_runs` | `pitcher_earned_runs` | Starter ERA splits + opposing lineup quality |
| `pitcher_walks` | `pitcher_walks` | Starter BB/9 blended already in feature set |

**Implementation:** Add keys to `ODDS_API_PLAYER_PROP_MARKETS`. The existing `_extract_odds_api_event_prop_rows` loop handles all of these automatically — it iterates all bookmaker → market → outcome groups. No schema change needed (`player_prop_markets.market_type` is VARCHAR(32), sufficient).

## Phase 2 — Umpire Data (Free, High Signal)

**Source:** UmpScorecards — `https://umpscorecards.com/api/` (free, no authentication)

### What to Fetch
- Per-game plate umpire assignment (linked to MLB game ID via Retrosheet/UmpScorecards)
- Umpire season averages: `called_strike_rate`, `k_rate_boost`, `bb_rate_boost`, `favor_home`, `run_value_per_game`

### New Ingestor: `src/ingestors/umpire.py`
Fetches today's ump assignments. Upserts into new `umpire_assignments` table. Follows same pattern as `src/ingestors/weather.py`.

### DB Migration: `022_umpire_assignments.sql`
```sql
CREATE TABLE IF NOT EXISTS umpire_assignments (
    assignment_id   BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    game_id         BIGINT NOT NULL,
    game_date       DATE NOT NULL,
    umpire_name     TEXT NOT NULL,
    ump_k_rate_adj  NUMERIC(6,4),   -- deviation from league-avg K rate
    ump_bb_rate_adj NUMERIC(6,4),   -- deviation from league-avg BB rate
    ump_run_value   NUMERIC(6,4),   -- runs above/below average per game
    source_name     TEXT DEFAULT 'umpscorecards',
    snapshot_ts     TIMESTAMPTZ NOT NULL,
    created_at      TIMESTAMPTZ DEFAULT now(),
    UNIQUE (game_id, snapshot_ts)
);
```

### Feature Integration
- `src/features/strikeouts_builder.py` — join `ump_k_rate_adj` on `game_id`; a tight-zone ump can swing K/9 by 10-15%
- `src/features/totals_builder.py` — join `ump_run_value`; a generous-zone ump adds ~0.5 runs/game to the total
- `src/utils/cli.py` — add umpire ingestion step to the "Update Lineups & Markets" pipeline action

## Phase 3 — New Model: Total Bases (`batter_total_bases`)

Follows the same architecture as `src/models/train_hits.py` but targets `actual_total_bases` instead of `actual_hit`.

### Feature Set (extends `src/features/hits_builder.py`)
- `xwoba_14g`, `xslg_14g` (add xSLG pull to batter_statcast ingestor), `iso_14g`, `hard_hit_pct_14g`
- `lineup_slot`, `bats` (L/R/S), platoon advantage vs opposing starter
- Opposing starter `xwoba_blended`, `hard_hit_pct_blended`
- Park factor (HR-friendly parks inflate TB)
- `game_total_line` (higher-scoring games → more TB opportunities per PA)
- `ump_k_rate_adj` (Phase 2 prerequisite for full accuracy)

### New Files
- `src/features/total_bases_builder.py` — extend `hits_builder`, add ISO/xSLG features
- `src/models/train_total_bases.py` — mirrors `train_hits.py`, target = actual TB
- `src/models/predict_total_bases.py` — mirrors `predict_hits.py`

### DB Migration: `023_total_bases_predictions.sql`
```sql
ALTER TABLE player_batting_daily
ADD COLUMN IF NOT EXISTS total_bases SMALLINT;   -- backfilled from existing box score data

CREATE TABLE IF NOT EXISTS predictions_total_bases (
    prediction_id       BIGINT GENERATED BY DEFAULT AS IDENTITY PRIMARY KEY,
    game_id             BIGINT NOT NULL,
    game_date           DATE NOT NULL,
    player_id           BIGINT NOT NULL,
    prediction_ts       TIMESTAMPTZ NOT NULL,
    model_name          TEXT NOT NULL,
    model_version       TEXT NOT NULL,
    predicted_tb        NUMERIC(5,2) NOT NULL,
    over_probability    NUMERIC(6,4),
    market_line         NUMERIC(4,1),
    edge                NUMERIC(6,4),
    created_at          TIMESTAMPTZ DEFAULT now(),
    UNIQUE (game_id, player_id, prediction_ts, model_name, model_version)
);
```

## Implementation Order (Recommended)

```
Phase 1A → Phase 1B → Phase 2 (DB + ingestor) → Phase 3 (DB + backfill) → Phase 4 → Phase 3 (train) → Phase 5
```

**Quickest win (~1 hour):** Phase 1A + 1B — add market keys to `market_totals.py`. No new DB schema, no new model, immediate value on next pipeline run.

**Biggest long-term edge:** Phase 2 (Umpire) + Phase 4 umpire feature — this is the most underpriced signal in public MLB models. A tight-zone umpire on a high-K starter is a systematic edge most books don't fully price in.

## Recommendation: Which Additional Market Lines Are Worth It?

Open up more market lines, but do it selectively. The right question is not "can we ingest it?" but "can we produce a stable edge from the features we already trust?"

### Tier 1 — Add Next

These fit the current architecture best and should create more dialed-in picks without forcing a brand-new modeling framework.

| Market | Recommendation | Why |
|---|---|---|
| `first_five_spread` | **Yes, once live rows exist** | We already model first-5 run environment and first-5 win context. Extending from first-5 moneyline/total to spread is a natural next step. |
| `first_five_team_total_home` / `first_five_team_total_away` | **Yes, once live rows exist** | These are even closer to the current expected-run split than full-game team totals because they are mostly starter-driven. |
| `first inning runs` (`YRFI` / `NRFI`) | **Maybe, but after first-5 spread/team totals** | Conceptually attractive and marketable, but it needs better first-time-through-order and top-of-lineup context to avoid becoming a high-variance novelty lane. |

### Tier 2 — Only After More Feature Work

These could eventually be useful, but they need better precision and should not be treated as immediate board-pick candidates.

| Market | Recommendation | Why |
|---|---|---|
| `first inning result` (home/away/tie) | **Later** | This is basically a compressed YRFI/NRFI plus side selection problem. The payoff can look attractive, but variance is high and one inning is a very noisy target. |
| `inning results` beyond the 1st | **No for now** | Once you get beyond the 1st inning, pitcher removal, bullpen timing, and lineup turn state create too much path dependence for the current product. |

### Tier 3 — Avoid for Now

These are the kind of markets that look fun but are more likely to dilute trust than improve pick quality.

| Market | Recommendation | Why |
|---|---|---|
| `winning margin bands` | **No for now** | Margin bands are a thin-probability, high-variance distribution tail. They need a much sharper full-run-distribution model than we currently have. |
| `4-way winning margin` | **No for now** | Same problem as bands, plus more bucket fragmentation. This is the opposite of a reliable board market. |

### Product Rule

New market types should be split into two buckets:

1. **Board candidates**: markets we trust enough to compete for top-5 green picks.
2. **Exploratory markets**: markets we expose in detail pages or secondary views, but do not let onto the main board until they prove stable.

Right now, `first_five_spread` and `first_five_team_totals` look like future **board candidates**. `YRFI/NRFI` is more likely an **exploratory market** first. Winning margins and inning-result exotics should stay out of the main recommendation surface.

### Practical Next Step

1. Add distinct ingestion support for `first_five_spread` and `first_five_team_total` market rows.
2. Extend the shared selector/grader in `src/utils/best_bets.py` only after those rows are flowing reliably.
3. If we want to test `YRFI/NRFI`, build it as a separate experimental lane first rather than mixing it into the main board.
